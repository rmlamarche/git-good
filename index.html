Overview

What's git?

  • Version Control System (VCS)
    • Track your code history over time
    • "branch" off in multiple directions
  • Issue Tracking
    • Keep a log of features, bugs, enhancements, problems, etc.
  • Collaboration
    • Work together on the same project
    • Avoid the "create a .zip and email it to me and I'll make changes and email it back to you" problem
  • Documentation
    • What does your code do?
    • How should other developers use your code?

What's the difference between git and GitHub?

  • Often used interchangably even though they're different
  • Git is the tool, GitHub is the service
    • In other words, git is how you interact with repositories like GitHub, GitLab, etc.

What do I need to git started?

  1. A Computer (Mac, Windows, Linux)
  2. Git - download and install from https://git-scm.com/downloads
  3. optional GitHub Desktop App - download and install from https://desktop.github.com/
  4. A GitHub Account
  5. A fork of https://github.com/rmlamarche/git-good

Exercise

1. Cloning the Repository

This is the process of downloading your repository from GitHub to your computer.

git clone https://github.com/rmlamarche/git-good.git

2. Making Changes

Use your favorite code editor here (VSCode, Atom, vim, vi, nano, gedit, notepad, notepad++, ... the list goes on)

echo "my name is your_name" >> README.txt

3. "Saving" your changes

Here, you're telling git that you like these changes and you want to save them. You're creating a timestamp - a moment in history for your code.

git commit -m "commit_message"

This is where you upload all of your commits to GitHub. You can commit multiple times before uploading them all.

git push origin master
Wait!
Now head to GitHub and check out your changes. Make sure they're there!

4. Whoops

You didn't mean to make those changes. Let's undo that!

git revert HEAD~

Frequently Asked Questions

Help! I made changes but I changed my mind them and I don't want to push them. What do I do?
$ git reset --hard
$ git pull
I'm on a different computer now, and it's telling me I have a merge conflict! How do I fix it?
A merge conflict happens when the changes you made conflict with what GitHub thinks the code should be. This happens often when working in teams, because while you are changing code, your teammate may also be making changes to the same code. You will have to go through and choose the changes you want to keep, and which ones you don't. This will be much easier using the GitHub Desktop Application.

Pull Requests

There's a whole lot more where that came from. A large topic we didn't cover in this workshop is Pull Requests (PRs). A PR is needed when you are working on more than one branch, and you want to merge (or pull) one of them into the other, making one big happy branch with all the code.

It's common practice to leave the master branch as the "stable" branch - your working copy, and to do all of your development on a different branch. When you're happy with the changes on the developement branch, you can go to GitHub and submit a PR to merge the development branch into the master branch. GitHub will tell you if there are any conflicts that you need to resolve, and then it will merge the branches for you.
